An engine where the LLM never authors a number.

Horizon3 turns a decision-maker's model, run on real market data, into an article, its charts, and its infographic — three renderings of one executed result. The intelligence is agentic; the numbers are not. This document maps the five LangGraph graphs, the GraphRAG model spine that feeds them, and the prompts that steer them.

The one idea

The number firewall

Every design decision in this repo defends a single rule from CLAUDE.md: the LLM never authors a number. Numbers are produced by executing a catalogued model on UMD data; the LLM only selects which model to run, designs how to show it, and narrates what it means. A companion rule keeps rendering deterministic — no diffusion model touches a numeric artifact.

The firewall is enforced structurally, not by asking nicely. The Chart Studio agents are shown a data profile — field names, types, ranges, the temporal span — and never the values. They emit an encoding that references field names with its data array forced empty; the compiler injects the real rows at render time.

"leave data EMPTY — the real rows are injected by the renderer; never author numbers."— the proposer node's own instruction, render/studio/nodes.py

System context · C4 level 1

What talks to what

Two triggers start work: a persona commissioned directly, or the ATS (article-trigger system) surfacing a candidate from central-bank calendars, data releases and standing pieces. The engine reaches one LLM (claude-opus-4-8, used for both reasoning and vision) and two data systems inside the external UMD platform: TimescaleDB/Postgres for the time series and executed outputs, and the Neo4j model spine for the proven-model vocabulary. The ChromaDB corpus is drawn as a dashed, design-time input — see the honesty note under the spine.

Component view · C4 level 2

Five graphs, one pattern

The subsystem is five LangGraph graphs that share a single house pattern: a state.py TypedDict, nodes.py of pure State → partial-State functions, and a graph.py that wires the StateGraph — every step traced to LangSmith. The article pipeline is the orchestrator; it invokes the selector, studio and judge as sub-graphs. The ATS is a separate commissioning graph that decides what to write before the pipeline decides how.

The pipeline, end to end

One state, many renderings

The article graph threads a single ArticleState object through eight nodes. Its design intent, quoted in the source: charts, prose, and the dashboard are projections of ONE state rather than three artifacts built from three unsynchronised sources. The draft node is an atomic best-of-N loop — write, ground, critique, keep-best — and the two reconcile_* nodes are the consistency stage the predecessor never had: they build the charts the prose actually names and derive the dashboard from the finished article.

The shared objectArticleState carries mat (executed models + numbers), brief (token menu, fact sheets, chart index, data-window firewall), outline, draft (best-of-N winner), bindings (section↔chart), cited_keys, the illustration/infographic PNGs, and result. Entry points: scripts/_regen_one.py (one persona, fast dev loop) and scripts/write_articles.py (all eight).

Deep dive · the signature agent

Chart Studio: choosing form as a reasoning problem

The Studio's premise, from its __init__: "chart choice is a reasoning problem, not a lookup table." The insight — what to show — is authored upstream; the Studio chooses the form: mark, encoding, and the editorial layer (reference lines, regime shading, direct labels) that separates craft from "a muppet with the FT". Seven nodes, two bounded loops, and — critically — a vision pass that looks at the rendered pixels.

The two loops terminate by construction: compile increments iterations every pass, and both conditional edges gate on iterations < max_iterations (default 3; the article pipeline calls it with 2). A failed compile or a failed revision degrades rather than crashes — a guaranteed-renderable fallback line keeps the critic and judge fed.

GraphRAG · the model spine

Retrieval that can only offer proven models

The live GraphRAG in this system is the Neo4j model spine. It follows the canonical three-stage shape — construct & index, retrieve, generate — but with a safety twist that is the whole point: a Model node carries executable:true only after it actually ran, alongside the run's points and as_of. The selector's retrieval returns that closed set, and the LLM chooses among it. It cannot pick a model that does not exist or cannot run, because it is never offered one.

"The LLM proposes — which models inform this decision is a judgement about meaning. The graph disposes — a model is offerable only if it is proven executable."— render/selector/nodes.py, paraphrased from the module docstring
Honesty note — read before citing "RAG"The Neo4j spine is the only retrieval that feeds an agent at runtime. The ChromaDB corpus under knowledge/ is design-time grounding only — it is queried by humans (and this very design process), not imported by any node in render/. A runtime literature-RAG is a planned future step (assessment §12), not a shipped feature. The docs must not imply otherwise.

The retrieval query

MATCH (m:Model {catalog:$cat, executable:true})
OPTIONAL MATCH (m)-[:PRODUCES]->(o:Output)
OPTIONAL MATCH (m)-[:RENDERS]->(v:Visualization)
RETURN m.id, m.name, m.family, m.method, m.points, m.as_of,
       collect(DISTINCT o.name + ' (' + o.unit + ') — ' + o.meaning) AS outputs,
       collect(DISTINCT v.insight) AS insights
ORDER BY m.id

The prompt catalogue

How the agents are instructed

Prompts here are inline string literals inside each nodes.py, paired with a Pydantic model via LangChain's .with_structured_output(Model). There are no classic function-calling tools — the structured-output schema is the tool surface, and each schema's Field(description=…) text is itself part of the prompt. Below, each key prompt is documented as a C4-code card: role, injected inputs, output schema, temperature — the SoTA way to convey a prompt without pasting a wall of text.

Recommendation, not yet doneSoTA treats every prompt as an immutable, versioned artifact referenced by a stable ID. These prompts live in code and are unversioned. A worthwhile next step is to extract them into a registry and give each a semver — flagged here, deliberately not refactored.

extract (grounding judge)

render/judge/nodes.py · _EXTRACT
Role
Find every sentence making a checkable assertion about an executed output and type it as one of five claim kinds (percentile, superlative, episode, regime, direction). It does not judge truth — arithmetic settles it. Precision over recall: a false accusation against true prose destroys trust in the check.
Injects
The offered outputs (the closed vocabulary — exact model_id/output pairs from Postgres), the prose (≤14k chars), known episodes, and any prior failure as feedback.
Output
Claims — later adjudicated against the executed series, never a model's memory.

The principles the prompts encode

  • LLM proposes / deterministic code disposes. Every graph splits a meaning judgement (LLM) from a fact or render (code): selector propose vs validate, studio propose vs compile, judge extract vs adjudicate.
  • Start from the message, not a chart you like. The framer names the communicative job before any mark is chosen.
  • Never author a number — enforced in code. data is stripped after every LLM emission; the profile shown upstream carries schema, not values.
  • Look at the pixels. A vision pass critiques the rendered chart and cross-checks the title against the plotted endpoint — the failure mode a text-only pipeline cannot catch.
  • Constrain to a closed, proven vocabulary. The selector and the judge can only bind to things that provably exist (executable models; offered outputs).
  • Feedback is an input, not a re-ask. A fixed-temperature model is nudged by changing the prompt (rejection reasons fed back), because re-asking reproduces the same output.

Where this design is grounded

The documentation form above follows the state of the art for agentic + GraphRAG systems: the C4-for-agentic-AI convention (draw the LLM explicitly; model gates as first-class nodes; render prompts as tables; iconize node kinds), the three-card framing (system / agent / spine), the canonical GraphRAG three-stage pipeline shape, and prompt-documentation practice (structured cards + encoded principles rather than raw strings).
